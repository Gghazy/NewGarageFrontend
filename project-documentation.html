1. نظرة عامة على المشروع

NewGarage هو نظام متكامل لإدارة مراكز فحص وصيانة السيارات. يوفر النظام إدارة كاملة لدورة حياة الفحص من لحظة استلام السيارة حتى تسليمها، مع نظام فوترة ومدفوعات متكامل.

الوظائف الرئيسية

  • إدارة الفحوصات: إنشاء أوامر فحص للسيارات وتتبعها عبر مراحل متعددة
  • مراحل الفحص المتعددة: 9 مراحل تشمل الحساسات، الطبلون، الهيكل الداخلي والخارجي، الديكور، الإكسسوارات، الميكانيكا، الإطارات، والتجربة الميدانية
  • نظام الفواتير: إنشاء فواتير تلقائية من الفحص، مع دعم الخصومات والضرائب والمدفوعات والاسترداد
  • إدارة العملاء: تسجيل بيانات العملاء وسياراتهم
  • إدارة الموظفين: تعيين الفنيين والمفتشين
  • إدارة الفروع: دعم الفروع المتعددة
  • نظام صلاحيات متقدم: أدوار وصلاحيات مفصلة لكل عملية
  • تقارير: تقارير الفحص، الإيرادات، ولوحة المعلومات
  • دعم اللغتين: عربي وإنجليزي مع دعم RTL

3. الهندسة المعمارية (Architecture)

يتبع المشروع نمط Clean Architecture مع مبادئ Domain-Driven Design (DDD) وCQRS (فصل القراءة عن الكتابة) عبر مكتبة MediatR.

طبقات النظام

🌐 Garage.Api — طبقة العرض (Presentation)

الـ Controllers، المصادقة، التوطين، نقاط النهاية REST API

⚙️ Garage.Application — طبقة التطبيق (Application)

الأوامر (Commands)، الاستعلامات (Queries)، المعالجات (Handlers)، خدمات التطبيق

💎 Garage.Domain — طبقة النطاق (Domain)

الكيانات، القيم، القواعد التجارية، الأحداث — قلب النظام

🔧 Garage.Infrastructure — البنية التحتية

قاعدة البيانات (EF Core)، المستودعات، المصادقة، الخدمات الخارجية

📋 Garage.Contracts — العقود

كائنات نقل البيانات (DTOs)، الطلبات (Requests)، الاستجابات (Responses) — مشتركة بين الطبقات

قاعدة أساسية: التبعيات تتجه للداخل فقط — الـ Domain لا يعتمد على أي طبقة أخرى. الطفرات (Mutations) تتم فقط عبر الـ AggregateRoot، والقراءة يمكن أن تتم عبر Projection مباشرة.

أنماط التصميم المستخدمة

النمطالاستخدام
CQRSفصل الأوامر عن الاستعلامات — كل عملية لها Command أو Query مستقل
Mediator (MediatR)توجيه الطلبات من الـ Controller للـ Handler المناسب
Repository Patternتجريد الوصول لقاعدة البيانات عبر IRepository<T> و IReadRepository<T>
Unit of Workضمان ذرية العمليات عبر IUnitOfWork
Aggregate Rootكل التغييرات تمر عبر جذر التجميع (مثل Examination، Invoice)
Smart Enumتعداد ذكي مع خصائص إضافية (مثل Stage)
Value Objectكائنات القيمة مثل Money للمبالغ والعملات
Projectionتحويل الكيانات لـ DTOs عبر Expression Trees في EF Core

4.1 إدارة الفحوصات (ExaminationManagement)

هذا هو السياق الرئيسي ويحتوي على الـ Aggregate Root Examination.

الكيان الرئيسي: Examination (الفحص)

هو جذر التجميع (AggregateRoot) الذي يدير كل عمليات الفحص. يحتوي على:

  • بيانات المركبة: لقطة من بيانات السيارة (VehicleSnapshot)
  • بيانات العميل: مرجع للعميل (ClientReference)
  • بيانات الفرع: مرجع للفرع (BranchReference)
  • عناصر الفحص: قائمة الخدمات المطلوبة (ExaminationItems)
  • نتائج المراحل: نتيجة كل مرحلة فحص (StageResults)
  • المدفوعات: قائمة المدفوعات (Payments)
  • السجل: تاريخ العمليات (History)

4.2 إدارة الفواتير (InvoiceManagement)

الكيان الرئيسي: Invoice (الفاتورة)

جذر تجميع (AggregateRoot) يدير الفاتورة وعناصرها ومدفوعاتها. يحتوي على:

  • عناصر الفاتورة (InvoiceItem): الخدمات وأسعارها
  • المدفوعات (InvoicePayment): عمليات الدفع والاسترداد
  • السجل (InvoiceHistory): تتبع كل العمليات
  • الحسابات: الإجمالي، الخصم، الضريبة، الإجمالي شامل الضريبة

5.3 الأنماط الأساسية (Base Patterns)

BaseCommandHandler<TCommand, T>

الكلاس الأساسي لجميع معالجات الأوامر — يوفر دوال مساعدة Ok() و Fail() لإرجاع النتائج.

BaseQueryHandler<TQuery, T>

الكلاس الأساسي لمعالجات الاستعلامات.

IRepository<T>

واجهة المستودع للكتابة: AddAsync، UpdateAsync، SoftDeleteAsync

IReadRepository<T>

واجهة المستودع للقراءة: Query()، QueryTracking()، AnyAsync()

IUnitOfWork

وحدة العمل: SaveChangesAsync()، BeginTransactionAsync()

7. طبقة الـ Infrastructure (البنية التحتية)

قاعدة البيانات

  • Entity Framework Core مع SQL Server
  • تكوين الجداول عبر IEntityTypeConfiguration<T>
  • تطبيق التكوينات تلقائياً عبر ApplyConfigurationsFromAssembly
  • دعم Soft Delete (حذف ناعم)
  • حقول تدقيق تلقائية (CreatedAt, UpdatedAt)

المستودعات (Repositories)

  • IRepository<T> — للكتابة (Add, Update, SoftDelete)
  • IReadRepository<T> — للقراءة (Query, AnyAsync)
  • ILookupRepository<T> — للبيانات الأساسية

المصادقة والتفويض

  • JWT Bearer Token للمصادقة
  • [HasPermission] attribute للتحقق من الصلاحيات
  • [HasAnyPermission] للتحقق من أي صلاحية من مجموعة

الخدمات الإضافية

  • InvoiceHtmlRenderer — عرض HTML للفاتورة (للطباعة/المشاركة)
  • InvoiceNumberGenerator — توليد أرقام الفواتير
  • InvoiceSyncService — مزامنة الفواتير
  • ExaminationService — خدمات مساعدة للفحوصات
  • Localization عبر Resource Files (عربي/إنجليزي)

9. نظام الصلاحيات والمستخدمين

النظام يستخدم نموذج Role-Based Access Control (RBAC) مع صلاحيات مفصلة على مستوى كل عملية.

كيف يعمل النظام

1
المستخدم ينتمي لـ دور (Role) واحد أو أكثر
2
كل دور يحتوي على مجموعة صلاحيات (Permissions)
3
الصلاحيات تُخزن كـ Claims في الـ JWT Token
4
الـ [HasPermission] يتحقق من وجود الصلاحية قبل تنفيذ الطلب

10. سير العمل الأساسي (Workflow)

10.1 دورة حياة الفحص

1
إنشاء الطلب: يتم إدخال بيانات العميل والسيارة واختيار الخدمات المطلوبة ← حالة Draft
2
بدء الفحص: يبدأ الفني العمل على المراحل ← حالة InProgress
3
تنفيذ المراحل: يمر الفحص على 9 مراحل (حسب الخدمات المختارة):

كل مرحلة تتضمن:

  • فحص أجزاء محددة وتسجيل حالتها (سليم، يحتاج صيانة، تالف...)
  • تسجيل المشاكل/الملاحظات لكل جزء
  • حفظ النتائج — يتم عبر الـ AggregateRoot (Examination)
4
إكمال الفحص: بعد الانتهاء من كل المراحل المطلوبة ← حالة Completed
5
إنشاء الفاتورة: يتم إنشاء فاتورة تلقائياً من الخدمات المسجلة
6
الدفع: يتم تسجيل المدفوعات على الفاتورة
7
التسليم: تسليم السيارة للعميل ← حالة Delivered

10.2 دورة حياة الفاتورة

1
إنشاء: تُنشأ من الفحص تلقائياً أو يدوياً ← حالة Issued
2
إضافة دفعة: يتم تسجيل دفعة بمبلغ وطريقة دفع
3
اكتمال الدفع: عند دفع كامل المبلغ ← حالة Paid

سيناريو تعديل سعر الخدمة

زيادة السعر: يتم إنشاء فاتورة Adjustment بالفرق ← يدفع العميل الفرق

تخفيض السعر: يتم إنشاء فاتورة Refund بالفرق ← يُسجل استرداد (Payment) على فاتورة الـ Refund لتسجيل رد المبلغ للعميل

11. نظام الفواتير والمدفوعات

هيكل الفاتورة

  • العميل: اسم ورقم هاتف
  • الفرع: الفرع المسؤول
  • العناصر (Items): الخدمات مع السعر لكل وحدة والإجمالي
  • الحسابات:
    • الإجمالي الفرعي (SubTotal)
    • الخصم (DiscountAmount)
    • نسبة الضريبة (TaxRate) ومبلغ الضريبة (TaxAmount)
    • الإجمالي شامل الضريبة (TotalWithTax)
  • المدفوعات (Payments): كل دفعة بمبلغ وطريقة دفع وتاريخ

الصفحة المجمعة (Consolidated)

تعرض ملخص لكل فواتير فحص معين مع الإجمالي المدفوع والمتبقي، حيث يتم حساب TotalPaid وBalance مباشرة من بيانات المدفوعات في الـ Domain.
